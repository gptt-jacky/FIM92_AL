## 一、設計目標

定義刺針模擬器的**後座力觸發流程**，釐清硬體端與軟體端的職責切分，確保：

- 軟體端掌握所有場景狀態判定（故障、飛彈狀態等）
- 硬體端負責實體操作判定與後座力輸出
- 雙方透過**唯一一個訊號**（IO7）完成授權機制

## 二、訊號切分架構

```
┌─────────────────────────────────────┐     ┌──────────────────────────────────────────┐
│          軟體端（費曼圖）              │     │          硬體端（全球動力科技）              │
│                                     │     │                                          │
│  場景狀態判定：                       │     │  實體訊號讀取：                             │
│  - BCU 裝卸                         │     │  - IO1  IFF 開關                           │
│  - 保險開關                          │     │  - IO2  鎖定鍵          ─┐                 │
│  - IFF 確認                         │     │  - IOA3 BCU              │                 │
│  - 飛彈是否在架                      │     │  - IOA4 保險             │ 回報給軟體       │
│  - 系統故障模擬                      │     │  - IO5  瞄準模組         │                 │
│  - 操作順序 / 秒數判定               │     │  - IO6  板機            ─┘                 │
│                                     │     │                                          │
│  ┌─────────────────────┐            │     │  後座力觸發判定：                           │
│  │ 判定結果：            │            │     │  ┌──────────────────────────────────┐     │
│  │ 允許射擊 / 不允許射擊  │            │     │  │ 條件 1: IO7 = ON（軟體允許）       │     │
│  └──────────┬──────────┘            │     │  │ 條件 2: IO8 = 1 （後座力準備就緒）  │     │
│             │                       │     │  │ 條件 3: IO6 + IO2 持續按壓 1 秒    │     │
│             ▼                       │     │  │ → 三者同時成立 → 觸發後座力         │     │
│  WebSocket: {"io7":"A1"} 允許        │     │  └──────────────────────────────────┘     │
│  WebSocket: {"io7":"A0"} 不允許      │     │                                          │
│             │                       │     │  後座力狀態回報：                           │
│             │     WebSocket          │     │  - IO8 = 1  準備就緒（可觸發）             │
│             └───────────────────────┼────►│  - IO8 = 0  觸發後冷卻中（不可觸發）        │
│                                     │◄────┼  IO 狀態字串即時回傳                       │
└─────────────────────────────────────┘     └──────────────────────────────────────────┘
```

## 三、後座力觸發流程

```
軟體端（費曼圖）                              硬體端（全球動力科技）
    │                                            │
    │  VR 場景中：                                │  持續讀取 IO 狀態並回報
    │  使用者依序操作                              │  （IO1~IO6, IO8 → WebSocket）
    │  BCU → 保險 → IFF...                       │
    │                                            │
    │  判定：操作正確、允許射擊                      │
    │                                            │
    ├──── {"io7":"A1"} ─────────────────────────►│  收到允許（IO7 = ON）
    │     （開保險）                               │
    │                                            │  檢查三條件：
    │                                            │  ✓ IO7 = ON（軟體已允許）
    │                                            │  ✓ IO8 = 1 （後座力準備就緒）
    │                                            │  ✓ IO6(板機) + IO2(鎖定) 持續 1 秒
    │                                            │
    │                                            │  ═══ 觸發後座力 ═══
    │                                            │
    │                                            │  IO8 → 0（機構動作，脫離就緒位置）
    │◄─── io 字串: IO8=0 ───────────────────────│  回報：後座力冷卻中
    │                                            │
    │  讀到 IO8=0                                 │  機構復位中（約 10 秒）
    │  → 知道已觸發                               │
    │                                            │
    │                                            │  機構復位完成
    │◄─── io 字串: IO8=1 ───────────────────────│  回報：後座力準備就緒
    │                                            │
    │  讀到 IO8=1                                 │
    │  → 可發出下一次允許                          │
    │                                            │
    ├──── {"io7":"A0"} ─────────────────────────►│  收回允許（IO7 = OFF）
    │     （關保險）                               │  → 按板機無效
    │                                            │
```

## 四、Tag A（刺針本體）IO 腳位定義

| 位置 | 腳位 | 方向 | 訊號名稱 | 說明 |
|------|------|------|----------|------|
| `[0]` | IO1 | 輸入 | IFF 開關 | 回報給軟體 |
| `[1]` | IO2 | 輸入 | 鎖定鍵 | **硬體觸發判定用** + 回報給軟體 |
| `[2]` | IOA3 | 輸入 | BCU | 回報給軟體 |
| `[3]` | IOA4 | 輸入 | 保險 | 回報給軟體 |
| `[4]` | IO5 | 輸入 | 瞄準模組 | 回報給軟體 |
| `[5]` | IO6 | 輸入 | 板機 | **硬體觸發判定用** + 回報給軟體 |
| `[6]` | IO7 | **輸出** | 軟體允許訊號 | 費曼圖透過 WebSocket 控制（保險開關） |
| `[7]` | IO8 | 輸入 | 後座力準備 | **硬體回報**：1=就緒 / 0=冷卻中 |

## 五、WebSocket 指令對照

### 軟體端 → 硬體端（控制）

| 指令 | 語義 | 說明 |
|------|------|------|
| `{"io7":"A1"}` | **允許觸發**（開保險） | 硬體收到後開始接受板機操作 |
| `{"io7":"A0"}` | **不允許觸發**（關保險） | 硬體忽略板機操作 |

### 硬體端 → 軟體端（狀態回報）

硬體端持續以 ~128 msg/sec 推送 JSON，軟體端從 `io` 字串讀取狀態：

```json
{"trackers":[{"tag":"A","px":...,"py":...,"pz":...,"rx":...,"ry":...,"rz":...,"rw":...,"io":"00110100"}]}
```

| io 位置 | 軟體端需關注的欄位 | 用途 |
|---------|-------------------|------|
| `[0]` IO1 | IFF 開關 | 判定操作流程 |
| `[1]` IO2 | 鎖定鍵 | 判定操作流程 |
| `[2]` IOA3 | BCU | 判定操作流程 |
| `[3]` IOA4 | 保險 | 判定操作流程 |
| `[4]` IO5 | 瞄準模組 | 判定操作流程 |
| `[5]` IO6 | 板機 | 判定操作流程 |
| `[6]` IO7 | 允許狀態回讀 | 確認目前是否已開保險 |
| `[7]` IO8 | **後座力準備** | **1=可再次允許 / 0=冷卻中** |

## 六、故障與異常情境處理

| 情境 | 負責方 | 處理方式 |
|------|--------|----------|
| 飛彈未離架 | 軟體（費曼圖） | 不送 `{"io7":"A1"}`，或送 `{"io7":"A0"}` 收回 |
| 飛彈掉落 | 軟體（費曼圖） | 送 `{"io7":"A0"}` 收回允許 |
| 系統判定不可射擊 | 軟體（費曼圖） | 不送允許 |
| 操作順序錯誤 | 軟體（費曼圖） | 軟體內部判定，不送允許 |
| 操作超時（如 BCU 逾時） | 軟體（費曼圖） | 軟體內部計時判定 |
| 後座力機構未復位 | 硬體（全球） | IO8=0，硬體自動擋（條件不滿足不會觸發） |

**核心原則：** 所有場景邏輯與故障判定皆由軟體端處理，硬體端不需要區分故障類型。軟體唯一送出的訊號就是「允許 / 不允許」，硬體只看這一個開關。

## 七、頭盔震動器（Tag B）現況

| 項目 | 現狀 | 說明 |
|------|------|------|
| 控制方式 | `{"io7":"B1"}` 小震動 / `{"io8":"B1"}` 大震動 | 軟體端直接控制 |
| 硬體配置 | IO7 + IO8 雙 Output | IO-only 裝置，無定位功能 |
| 觸發邏輯 | 軟體直接送指令，無硬體判定 | 與刺針不同，震動器由軟體端全權控制 |

## 九、待會議確認事項

1. **IO7 作為「允許訊號」是否可行？** 沿用現有 `{"io7":"A1/A0"}` 指令，語義從「直接控制後座力」改為「允許 / 不允許觸發」
2. **硬體判定邏輯（IO6+IO2 持續 1 秒）是否合理？** 或是需要調整秒數 / 按鍵組合
3. **操作順序與秒數判定全由軟體端負責，是否符合貴方規劃？**
4. **故障情境（飛彈未離架、掉落等）全由軟體端判定後以「不允許」處理，是否足夠？** 或是否有額外狀態需要回傳給硬體端？
5. **頭盔震動器維持軟體直接控制，是否有其他需求？**
